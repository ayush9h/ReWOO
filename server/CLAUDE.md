## Project Overview

ReWOO (Reasoning Without Observation) is a LangGraph-based implementation of the plan-and-execute paradigm with Human-In-The-Loop (HITL) capabilities.

The system decouples planning from execution. A planner first generates a structured plan, and an executor then runs the required steps with dependency resolution. This allows parallel tool execution and reduces token usage compared to traditional ReAct-style patterns.

## Key Components

### AgentState

The global shared state object that flows through LangGraph nodes.

It contains:

- User query
- Generated plan
- Execution evidence
- Final response

### Plan

A collection of steps representing the execution strategy generated by the planner.

### Step

An individual unit of work.

A step contains:

- Instruction
- Tool metadata
- Dependencies
- Execution status
- Evidence

Supported statuses:

```text
pending
running
success
failed
pending_human_approval
```

### Evidence

A container for tool inputs and outputs.

Evidence usually includes:

- Tool name
- Tool input
- Tool output
- Execution content
- Failure or rejection reason, if applicable

## LangGraph Nodes

### planner_node

Transforms the user query into a structured `Plan` using an LLM.

Responsibilities:

- Understand the user request
- Break the request into executable steps
- Select tools where needed
- Assign step dependencies
- Return a structured plan

### executor_node

Executes ready steps asynchronously.

Responsibilities:

- Identify steps whose dependencies are complete
- Execute independent steps in parallel where possible
- Call tools from the registry
- Update step status
- Store evidence
- Detect tools that require human approval

### hitl_node

Handles Human-In-The-Loop approval using LangGraph interrupts.

Responsibilities:

- Find the step with `pending_human_approval`
- Raise an interrupt
- Send approval metadata to the frontend
- Resume execution based on user approval or rejection

Approval payload shape:

```json
{
  "approved": true
}
```

Rejection payload shape:

```json
{
  "approved": false
}
```

Important: keep the current interrupt-based HITL implementation unless explicitly asked to refactor it.

### summarizer_node

Produces the final response from collected evidence.

Responsibilities:

- Read completed step evidence
- Handle failed or rejected steps gracefully
- Generate the final user-facing response

## Tool Registry

The `TOOL_REGISTRY` is the central source of truth for tools.

Each tool can define:

- `fn`: function implementation
- Input schema
- Output schema
- Whether human approval is required
- Whether the tool can be chained
- `next_tool_call` for automatic chaining

Example chaining pattern:

```text
LeaveTool -> OutOfOfficeTool
```

## Data Flow

1. User sends a query through the frontend.
2. Frontend sends the message over WebSocket.
3. Backend receives the query.
4. `planner_node` generates a structured plan.
5. Plan is streamed to the frontend.
6. `executor_node` identifies ready steps.
7. Ready steps are executed.
8. If a step requires approval, `hitl_node` raises an interrupt.
9. Frontend displays the approval request.
10. User approves or rejects.
11. Backend resumes graph execution.
12. `summarizer_node` creates the final response.
13. Final response is streamed back to the frontend.

## Dependency Resolution

Use `get_ready_steps` to determine executable steps.

A step is ready when:

- Its status is `pending`
- Every step in its `depends_on` list has status `success`

This enables parallel execution of independent steps while preserving ordering for dependent steps.

## Human-In-The-Loop Pattern

HITL is implemented using LangGraph interrupts.

Expected approval flow:

1. A tool requiring approval is reached.
2. The related step is marked `pending_human_approval`.
3. `hitl_node` triggers an interrupt.
4. Backend sends an `interrupt` event over WebSocket.
5. Frontend stores the pending approval.
6. User approves or rejects.
7. Frontend sends a resume message with an `approved` boolean.
8. Backend resumes graph execution.

Rejected approvals must not route back into the same approval state endlessly.

When a user rejects a tool call:

- Mark the step as `failed`
- Store rejection reason in evidence
- Continue to remaining executable steps if possible
- Otherwise route to summarization
- Avoid recursion-limit loops

## Important Implementation Notes

### Preserve Separation of Concerns

Keep responsibilities isolated:

- Planner creates plans
- Executor runs steps
- HITL node handles approval
- Summarizer writes the final response
- Tools perform external actions

Do not merge these responsibilities unless there is a strong reason.

### Avoid ReAct-Style Refactors

This project is based on the ReWOO plan-and-execute pattern.

Do not convert it into a ReAct loop unless explicitly requested.

### Preserve HITL Interrupts

The HITL flow depends on LangGraph interrupts.

Do not replace interrupts with custom polling or manual state flags unless explicitly requested.

### Handle Rejections Safely

A rejected approval should never cause an infinite loop.

Make sure rejected steps are no longer eligible for approval handling.

### Handle Dependency Deadlocks

If a step depends on a failed step, it may never become ready.

Handle this by:

- Marking dependent steps as failed or skipped
- Allowing partial summarization
- Explaining unavailable results in the final response

## Common Pitfalls

### Raw Boolean Resume Payload

The HITL node expects an object with an `approved` key.

Avoid passing:

```json
true
```

Use:

```json
{
  "approved": true
}
```

### Rejection Recursion

If rejected steps remain `pending_human_approval`, LangGraph may repeatedly return to `hitl_node` until the recursion limit is reached.

Fix by setting rejected steps to `failed`.

### Duplicated Frontend State

Avoid duplicating backend execution logic in the frontend.

The frontend should render state and send user actions. The backend should own workflow execution.

### Unstable WebSocket Payloads

Avoid changing event shapes without updating both sides.

## Preferred Change Strategy

When modifying this project:

1. Identify the affected layer.
2. Keep changes small and production-safe.
3. Preserve the existing architecture.
4. Update schemas before changing node behavior.
5. Keep WebSocket event shapes stable.
6. Add explicit error handling.
7. Handle HITL approval and rejection paths.
8. Avoid broad rewrites unless requested.
